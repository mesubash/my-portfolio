# From Kathmandu to Code

I grew up in a country where load-shedding was a daily reality. Where internet connections were measured in kilobytes, not megabytes. Where the idea of building software for a living felt like something that happened *somewhere else*.

But Kathmandu has a way of teaching you something that no bootcamp or university can: **how to build with constraints.**

## Learning in the Margins

My early programming education happened in fragments. Downloaded tutorials saved for offline reading. Code written in notebooks during power cuts. Projects built on machines that would crash if you opened too many Chrome tabs.

These constraints weren't romantic — they were frustrating. But they forced me to think differently about efficiency, about what's essential, and about the value of every resource.

When you've waited 20 minutes for a page to load, you understand viscerally why performance matters. When you've lost work to a power outage, you understand why saving state is important. When you've debugged without Stack Overflow, you understand why reading documentation matters.

## The First Real Project

I remember the first time I built something that someone else actually used. It was a small utility — nothing impressive by any standard. But watching someone interact with code I'd written, seeing it solve a real problem, changed something in me.

It wasn't about the technology. It was about the **agency**. The realization that I could create tools that didn't exist before. That I could solve problems that affected real people. That the gap between *idea* and *implementation* was just effort and learning.

## Building Yugo

Working on Yugo — a QR-based fare collection system — brought everything full circle. Here was a problem rooted in Kathmandu's daily life: chaotic public transport, cash-only payments, no accountability in fare collection.

The technical challenges were real: unreliable connectivity, diverse hardware, users with varying tech literacy. But these were the kinds of constraints I'd grown up with. They weren't obstacles — they were design parameters.

We built a system that worked offline-first, synced when connected, and failed gracefully when it couldn't. Not because we read about these patterns in a textbook, but because we understood the environment we were building for.

## What Nepal Taught Me

Every engineer's origin story shapes their engineering philosophy. Mine taught me:

- **Resourcefulness over resources** — You don't need the best tools to build the best solutions
- **Resilience by design** — Systems should work in imperfect conditions because conditions are always imperfect
- **Empathy in engineering** — The best technical decisions come from understanding the people who'll use what you build
- **Patience with progress** — Good things take time, and that's okay

## Looking Forward

I'm still based in Kathmandu. The load-shedding is mostly gone now. The internet is faster. The tech community is growing. And every day, more engineers here are building things that matter.

I write code now with the same energy I had when I was downloading tutorials on a 2G connection: with gratitude for the opportunity and urgency about making the most of it.

The journey from Kathmandu to code wasn't a departure. It was a discovery that the skills I needed were already here — taught by a city that never made anything easy, and was better for it.
